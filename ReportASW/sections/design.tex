\section{Design}

Questa sezione descrive la progettazione architetturale di KompozeR in termini di strutturazione del dominio, separazione delle responsabilita e coerenza tra modello e implementazione. L'obiettivo e' mostrare come requisiti, bounded context, servizi, contratti applicativi e persistenza siano stati allineati in un'unica soluzione tecnica.

Nel seguito si fa riferimento anche agli artefatti di progettazione aggiornati in \texttt{utilities} (bounded context, mappa endpoint, definizione DB e domain DTO), riallineati rispetto allo stato corrente del codice.

\subsection{Obiettivi di design}

Le principali decisioni progettuali sono state prese con l'intento di raggiungere i seguenti obiettivi:

\begin{itemize}
	\item separazione chiara tra frontend, gateway e servizi backend;
	\item indipendenza dei moduli del dominio (autenticazione, catalogo, configurazione CAD, carrello, ordini, notifiche, chatbot, reporting);
	\item favorire manutenibilita', evoluzione incrementale e chiarezza delle responsabilita';
	\item supportare un'esperienza utente interattiva, con feedback rapidi su configurazione, prezzo e notifiche;
	\item centralizzare autenticazione e controllo accessi nel gateway, riducendo duplicazione di logica nei microservizi.
\end{itemize}

\subsection{Visione architetturale}

KompozeR e' stato progettato come una Single Page Application che comunica con un backend organizzato in piu' microservizi con architettura esagonale che aderisce al modello REST (Representational State Transfer). 
Il frontend concentra la logica di presentazione e di interazione con l'utente, mentre il backend espone servizi applicativi separati per area di responsabilita'.

Dal punto di vista logico, l'architettura si articola in tre livelli:

\begin{itemize}
	\item frontend web, responsabile dell'interazione con utenti finali e amministratori;
	\item API Gateway, punto di ingresso unico per richieste HTTP e handshake real-time;
	\item microservizi backend, ciascuno focalizzato su un sottoinsieme specifico del dominio.
\end{itemize}

\subsection{Suddivisione in microservizi}

\subsubsection{Bounded Context}

\begin{enumerate}
\item \textbf{Authentication (support)}
\begin{itemize}
\item Responsabilita: gestione identita, login, registrazione, sessioni e ruoli.
\item Possiede: utenti (\texttt{users}) e sessioni (\texttt{sessions}).
\item Non possiede: catalogo, configurazioni, carrello, ordini, notifiche, chat, reporting.
\end{itemize}

\item \textbf{Catalog (support)}
\begin{itemize}
\item Responsabilita: gestione componenti, prezzo, disponibilita e ricerca catalogo.
\item Possiede: componenti (\texttt{components}).
\item Pubblica eventi su Redis (\texttt{catalog:events}) quando cambiano prezzo/disponibilita.
\end{itemize}

\item \textbf{CAD (core)}
\begin{itemize}
\item Responsabilita: ciclo di vita della configurazione scaffalatura.
\item Possiede: configurazioni (\texttt{configurations}) con stato, ambiente, categoria, piano colonne, design e BOM.
\item Dipende da Catalog per regole/componenti e da Cart/Notification per finalizzazione e sottoscrizioni.
\end{itemize}

\item \textbf{Cart (core)}
\begin{itemize}
\item Responsabilita: gestione carrello utente e checkout.
\item Possiede: carrelli (\texttt{carts}) con item snapshot (sku, prezzo unitario, quantita, totale riga).
\item Reagisce agli eventi catalogo per rimuovere/ripristinare articoli e riallineare prezzi.
\end{itemize}

\item \textbf{Order (core)}
\begin{itemize}
\item Responsabilita: persistenza ordini e transizioni di stato.
\item Possiede: ordini (\texttt{orders}) con stati \texttt{SUBMITTED}, \texttt{DONE}, \texttt{CANCELLED}.
\end{itemize}

\item \textbf{Notification (core)}
\begin{itemize}
\item Responsabilita: sottoscrizioni utente e notifiche in-app.
\item Possiede: notifiche (\texttt{notifications}), sottoscrizioni (\texttt{notificationSubscriptions}), eventi processati (\texttt{notificationEvents}).
\item Consuma eventi catalogo da Redis e risolve impatti su contesti CAD/CART/SUBSCRIPTION.
\end{itemize}

\item \textbf{Chatbot (core)}
\begin{itemize}
\item Responsabilita: sessioni chat e messaggistica contestuale.
\item Possiede: sessioni (\texttt{chatSessions}) e messaggi (\texttt{chatMessages}).
\item Dipende da Catalog e CAD per risposte contestuali.
\end{itemize}

\item \textbf{Reporting (support)}
\begin{itemize}
\item Responsabilita: trend ordini per area amministrativa.
\item Non possiede un DB dedicato nel setup corrente: legge la collection \texttt{orders} (orderdb) in sola lettura.
\end{itemize}
\end{enumerate}

\subsubsection{Context Map (stato corrente)}

\begin{itemize}
\item Frontend $\rightarrow$ API Gateway: entry point unico per API e websocket.
\item API Gateway $\rightarrow$ Authentication: autenticazione/ruoli/sessioni.
\item API Gateway $\rightarrow$ Catalog/CAD/Cart/Order/Notification/Chatbot/Reporting: proxy REST.
\item Catalog $\rightarrow$ Notification: eventi \texttt{PRICE\_CHANGED} e \texttt{AVAILABILITY\_CHANGED} su Redis.
\item Catalog $\rightarrow$ Cart: eventi Redis per riallineamento carrello.
\item CAD $\rightarrow$ Cart: finalizzazione configurazione e sincronizzazione BOM.
\item CAD $\rightarrow$ Notification: creazione sottoscrizioni prodotto in fase di finalize.
\item Order $\rightarrow$ Reporting: reporting aggrega trend da \texttt{orders}.
\item Chatbot $\rightarrow$ Catalog/CAD: recupero contesto per risposte.
\end{itemize}

\subsection{API Gateway e comunicazione tra componenti}

Il gateway semplifica la comunicazione client-backend, verifica JWT e controllo accessi. 
Il frontend interagisce con un unico endpoint e non dipende da URL o policy specifiche dei singoli servizi.

Il flusso e' il seguente:

\begin{itemize}
	\item il client invia \texttt{Authorization: Bearer <token>} al gateway;
	\item il gateway valida il token e propaga ai servizi header di identita' (es. \texttt{X-User-Id}, \texttt{X-User-Role}, \texttt{X-Session-Id});
	\item il routing proxy inoltra la richiesta al servizio corretto (auth, catalog, cad, cart, orders, notifications, chatbot, reports).
\end{itemize}

Le interazioni principali usano REST. Le funzionalita' real-time usano Socket.IO:

\begin{itemize}
	\item canale notifiche in-app per push di eventi utente;
	\item canale chatbot per invio/ricezione messaggi in tempo quasi reale.
\end{itemize}

Le notifiche dominio nascono da eventi catalogo (prezzo/disponibilita') pubblicati su Redis e consumati dal notificationService.

\subsection{Modello del dominio}

Dal punto di vista applicativo, il dominio ruota attorno a entita' centrali: utente, prodotto, configurazione, carrello, ordine, sessione chat, notifica. Le relazioni principali sono:

\begin{itemize}
	\item una \texttt{Configuration} appartiene a un utente ed e' vincolata a una categoria (TONDO, QUADRO, KUBE);
	\item la finalizzazione di una configurazione genera una BOM e aggiorna il carrello;
	\item il \texttt{Catalog} e' fonte autorevole di prezzo/disponibilita';
	\item cambi catalogo possono impattare i contesti utente (configurazioni finalizzate, carrelli, sottoscrizioni) e generare notifiche;
	\item il \texttt{Reporting} aggrega i dati ordine per intervallo temporale.
\end{itemize}

La persistenza segue ownership esplicita per contesto: ogni servizio possiede il proprio schema Mongo e non condivide accesso diretto al database altrui, con l'eccezione di proiezioni di sola lettura necessarie a risoluzione impatti notifiche.

\subsection{Sicurezza e controllo accessi}

La sicurezza applicativa si basa su JWT e distinzione dei privilegi per ruolo.

Le rotte pubbliche sono limitate a registrazione/login/guest. Tutte le altre richiedono token valido. Le operazioni amministrative (catalogo admin, reporting, gestione ordini admin) richiedono ruolo \texttt{ADMIN}.

Il gateway elimina eventuali header identita' inviati dal client e li reinserisce solo dopo verifica JWT, riducendo il rischio di spoofing.

Il modello di accesso implementato distingue chiaramente:

\begin{itemize}
	\item il ruolo del servizio di autenticazione;
	\item l'uso dei token per identificare l'utente lato backend;
	\item la distinzione tra utente guest, utente autenticato e amministratore;
	\item il fatto che il gateway rappresenta il primo punto di enforcement delle policy di accesso.
\end{itemize}

\subsection{Design Pattern utilizzati}
Nel codice sono stati applicati pattern architetturali e applicativi coerenti con la natura distribuita del sistema.

\begin{itemize}
\item \textbf{Microservices + API Gateway}: il backend e' suddiviso in servizi autonomi per bounded context, mentre il gateway funge da entry point unico e centralizza autenticazione, propagazione identita' e routing verso i servizi interni.

\item \textbf{Hexagonal Architecture (Ports \& Adapters)}: nei servizi backend la logica applicativa dipende da porte astratte (es. repository e provider), mentre dettagli infrastrutturali (Mongo, HTTP, Redis, Socket.IO) sono implementati come adapter. Questo riduce l'accoppiamento tra dominio e tecnologia.

\item \textbf{Use Case / Application Service}: le operazioni applicative sono modellate come classi di use case con metodo \texttt{execute} (ad esempio creazione sessione chat, invio messaggi, gestione sottoscrizioni, update stato ordini), separando orchestrazione applicativa e trasporto HTTP.

\item \textbf{Repository Pattern}: l'accesso alla persistenza e' incapsulato dietro interfacce repository; le implementazioni concrete Mongo sono sostituibili e testabili in isolamento.

\item \textbf{Adapter Pattern}: integrazioni esterne sono incapsulate in adapter dedicati. Un esempio rilevante e' il chatbot che usa adapter HTTP per recuperare contesto catalogo/configurazione senza dipendere direttamente dai dettagli dei servizi remoti.

\item \textbf{Event-Driven Integration (Pub/Sub)}: la propagazione di variazioni catalogo avviene tramite eventi Redis (es. prezzo/disponibilita'), consumati da Notification e Cart per aggiornamenti asincroni e disaccoppiati.

\item \textbf{Observer-style su canali realtime}: le componenti realtime basate su Socket.IO seguono un modello publish/subscribe con listener registrati lato client per nuovi messaggi, typing e notifiche push.

\item \textbf{Facade/BFF lato frontend services}: il frontend espone servizi applicativi (es. \texttt{chatbotService}, \texttt{catalogService}, \texttt{cadService}) come facciata verso API e socket, semplificando l'uso da parte delle view.

\item \textbf{Composable Pattern (Vue 3)}: la logica riusabile di dominio UI (catalogo, carrello, CAD, notifiche) e' estratta in composables, separando presentazione e orchestration.
\end{itemize}

L'utilizzo di questi pattern ha permesso di avere un sistema modulare e testabile, che permette anche una facile manutenzione e evoluzione. 
Questo ha permesso inoltre di mantenere single concerns per ogni modulo, riducendo l'accoppiamento tra le componenti.

\subsection{Organizzazione del frontend}

Il frontend e' stato realizzato come una SPA Vue di modo tale da garantire una separazione per responsabilita':

\begin{itemize}
	\item \texttt{views}: schermate applicative (auth, catalogo, CAD, carrello, configurazioni, notifiche, area admin);
	\item \texttt{services}: client HTTP e socket verso i backend services;
	\item \texttt{store}: stato trasversale (sessione, badge carrello, badge notifiche);
	\item \texttt{composables}: logica UI riusabile e orchestration dei flussi.
\end{itemize}

Le principali viste applicative sono:

\begin{itemize}
	\item area catalogo e accesso al configuratore;
	\item configuratore visuale della scaffalatura;
	\item area carrello e riepilogo richiesta d'ordine;
	\item area notifiche;
	\item interfaccia chatbot di supporto;
	\item area amministrativa per catalogo, gestione stati ordine e reportistica trend ordini.
\end{itemize}

La navigazione applica guardie di route coerenti con ruoli e tipo utente. In particolare, le viste admin sono riservate a ruolo \texttt{ADMIN}, mentre i flussi utente standard sono disponibili a utenti autenticati, inclusi guest tokenizzati.

\subsection{Design delle interfacce utente}

Il design delle interfacce e' stato orientato a ridurre la complessita' percepita durante la composizione della scaffalatura. Le principali scelte sono state guidate dall'esigenza di mostrare solo le azioni rilevanti, mantenere visibili prezzo e stato della configurazione e fornire supporto contestuale senza interrompere il flusso principale.

Scelte progettuali principali:

\begin{itemize}
	\item flusso CAD guidato a step (ambiente, categoria, piano colonne, design, finalizzazione);
	\item persistenza progressiva della configurazione ai passaggi confermati, con ripresa in sessioni successive;
	\item feedback immediato tramite toast, stati di loading ed error handling coerente;
	\item notifiche in-app non intrusive con badge e inbox dedicata;
	\item chatbot integrato nel flusso utente con risposte contestuali su catalogo;
	\item separazione netta tra viste utente e viste amministrative.
\end{itemize}

\subsection{Considerazioni finali di progetto}

Nel complesso, il design di KompozeR bilancia semplicita' d'uso lato utente e modularita' lato backend. 
L'architettura a servizi con gateway centralizzato, il frontend SPA organizzato per responsabilita' e l'uso combinato di REST e canali real-time forniscono una base coerente con gli obiettivi del progetto.

La soluzione e' inoltre estendibile: nuovi moduli applicativi possono essere aggiunti preservando separazione dei contesti, contratti di API e coerenza dei flussi utente.
